# DevMate 项目计划

## 1. 这份计划是干什么的

这份文件的目标不是写给评审系统看的，而是写给“人”看的。你可以把它当成这个项目的施工图，打开后就知道今天该做什么、哪些是硬性要求、哪些风险会卡住进度。

## 2. 项目一句话目标

做一个 AI 编程助手。它能理解用户需求，必要时联网搜索，必要时检索本地文档，然后生成或修改一个多文件代码项目，并把成功做法沉淀成可复用 Skills。

## 3. 当前仓库状态

目前已经完成的事情：

- Git 仓库已初始化
- 基础目录已经建好
- Python 包结构已经建好
- 配置文件模板已经补齐
- Docker 骨架文件已经补齐
- 中文计划和说明文档已经落盘

目前还没完成的关键事情：

- 本机缺少 Python 3.13
- 本机缺少 `uv`
- 还没有真实接入 LangChain、Tavily、LangSmith、ChromaDB
- 还没有真正实现 MCP Server、MCP Client、RAG、Skills

## 4. 开发目标拆分

这个项目建议拆成 6 个工作块：

1. 基础工程
2. MCP 搜索链路
3. RAG 文档检索
4. Agent 核心流程
5. Skills 复用机制
6. Docker 与交付包装

## 5. 4 天详细计划

### Day 1：把工程和环境打牢

目标：
先把所有“后面一定会用到的基础设施”搭起来，避免第二天开始写功能时才发现环境不对。

要完成的事：

- 安装 Python 3.13
- 安装 `uv`
- 用 `uv sync` 建立虚拟环境和依赖
- 确认 `pyproject.toml`、`config.toml`、`.gitignore` 可用
- 明确源码目录、测试目录、文档目录
- 选定最终技术路线：`LangChain + Tavily + ChromaDB + LangSmith`
- 先写一个最小 CLI 入口，保证项目可以启动

完成标准：

- `uv run devmate` 可以正常启动
- 配置能从 `config.toml` 读取
- Git 已连接到 GitHub 远程仓库
- 至少有 3 到 5 次清晰提交，不要一次性提交全部内容

建议提交节奏：

- `chore: initialize repository skeleton`
- `chore: add pyproject and config loading`
- `docs: add project plan and architecture notes`

### Day 2：打通 MCP 搜索链路

目标：
让 Agent 具备“需要时联网查资料”的能力。

要完成的事：

- 实现最小 MCP Server
- 暴露 `search_web` 工具
- 把 `search_web` 接到 Tavily API
- 确认传输方式是 Streamable HTTP
- 在主应用中实现 MCP Client
- 让 Agent 能通过 Client 调用搜索工具
- 做一次手动验证并记录日志

完成标准：

- 用户输入一个需要联网的请求后，Agent 会触发 `search_web`
- 日志里能看到搜索请求和返回结果
- 整个搜索流程不是写死在 Agent 里，而是通过 MCP 调用

建议提交节奏：

- `feat: add mcp search server skeleton`
- `feat: wire tavily search through mcp client`
- `test: add manual verification notes for web search`

### Day 3：完成 RAG、Agent 主流程和 Skills 基础

目标：
让 Agent 不仅会搜外网，还会查本地资料，并开始形成“可复用做法”。

要完成的事：

- 给 `docs/` 加入样例知识文档
- 做文档切分和索引
- 接入本地向量数据库
- 实现 `search_knowledge_base`
- 在 Agent 中统一接入 `search_web` 和 `search_knowledge_base`
- 设计系统提示词
- 设计 Skills 的存储格式和发现规则

完成标准：

- 一个需求请求能同时触发联网搜索和本地知识检索
- Agent 输出里能体现用了哪些知识来源
- 至少能把一类成功流程保存为 Skill 文件或 Skill 记录

建议提交节奏：

- `feat: add local knowledge ingestion`
- `feat: integrate rag retrieval into agent runtime`
- `feat: add initial skill registry structure`

### Day 4：完成端到端演示、Docker 和交付

目标：
把项目从“能开发”推进到“能交付”。

要完成的事：

- 跑通“徒步路线网站”演示场景
- 确认 Agent 会先搜网、再查本地文档、再规划文件
- 输出多文件项目结果
- 接入 LangSmith 或 LangFuse
- 补齐 `Dockerfile` 和 `docker-compose.yml`
- 完善 README
- 整理提交历史和交付说明

完成标准：

- `docker compose up` 能把项目拉起来
- 有一次完整的 Trace 可展示
- README 说明清楚如何配置、如何运行、如何验证
- GitHub 仓库可直接提交给面试官

建议提交节奏：

- `feat: add end-to-end hiking website workflow`
- `feat: add docker runtime for app and services`
- `docs: finalize delivery guide and verification steps`

## 6. 你每天实际应该怎么安排

如果你每天能投入 6 到 8 小时，建议这样分配：

- 上午前半：写核心代码
- 上午后半：手动验证
- 下午前半：补测试和日志
- 下午后半：补文档、提交 Git、记录当天结果

每天结束前一定做三件事：

1. 提交代码
2. 更新 README 或计划
3. 记录今天已验证通过了什么

## 7. 风险清单

最容易卡住你的点不是业务逻辑，而是环境和集成：

- `uv` 没装，第一天就会卡住
- Python 不是 3.13，后面会和题目要求不一致
- Tavily Key 没准备好，MCP 搜索无法联调
- LangSmith 或 LangFuse 没配置好，直接失去一票否决项
- Docker 最后一天才做，很容易来不及

## 8. 最低可交付版本

如果时间真的紧，最低也要确保这些东西都在：

- GitHub 仓库
- `pyproject.toml`
- `config.toml`
- 可运行入口
- MCP 搜索链路
- 基础 RAG
- Skills 最小实现
- Dockerfile
- docker-compose.yml
- README

## 9. 我建议你优先选的路线

为了在 4 天内稳稳做完，优先选这一套：

- Agent 框架：LangChain
- LLM：OpenAI 兼容接口
- 搜索：Tavily
- MCP：Streamable HTTP
- RAG：ChromaDB
- 可观测性：LangSmith
- 交互方式：CLI 优先，Web UI 后补

原因很简单：这条路线资料多、组合成熟、工程阻力最小。

## 10. 打开这个仓库时先看什么

第一次打开仓库，建议按这个顺序查看：

1. `README.md`
2. `项目计划.md`
3. `docs/architecture.md`
4. `config.toml`
5. `src/devmate/main.py`

这样你会先知道项目为什么存在，再知道今天该做什么，最后再看代码入口。
